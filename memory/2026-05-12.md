### 🎯 Next Proposal — #38: Autonomous Test Coverage & Threshold Enforcement

**Problem:** Test coverage can silently drift downward as new features added; no gate prevents erosion.

**Solution:** CI gate that fails when coverage drops below configured threshold; baseline auto-updates on `main`; trend tracking.

**Components:**
- Script: `automation/test-coverage-enforcer.cjs`
- Workflow: `.github/workflows/test-coverage.yml` (on PR to main)
- Docs: `docs/TEST-COVERAGE-ENFORCEMENT.md`
- State: `.hermes/memory/test-coverage/` (baseline.json, history.json, 90d trend)

**Behavior:**
- Run `npm test -- --coverage --silent` on PR-built site
- Parse Istanbul JSON (`coverage/coverage-final.json`)
- Compare current coverage vs baseline stored in repo
- Fail CI if:
  - Global coverage < configured minimum (default 80%)
  - Any file coverage drops >5% relative
  - Lines/branches/functions drop vs baseline
- Auto-update baseline on merges to `main` (only when coverage improves or stays within tolerance)
- Send Telegram summary on PR failures
- Track 90-day trend; alert on sustained degradation

**Config (env vars):**
- `COVERAGE_MIN_GLOBAL` (default: "80")
- `COVERAGE_MIN_FILES` (comma-separated "file:threshold" pairs)
- `COVERAGE_REGRESSION_PCT` (default: "5" — max allowed drop per file)
- `COVERAGE_UPDATE_BASELINE_ON_MAIN` (default: "true")

**Safety:** No external dependencies; uses Istanbul output; deterministic; baseline committed to repo.

**Standing:** Ready to implement autonomously per owner mandate.
