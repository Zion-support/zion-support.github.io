## 🎯 Next Proposed Improvement (#34): Field Performance & Real User Web Vitals Monitoring

**Motivation:** Synthetic Lighthouse tests ≠ real user experience. Need field Core Web Vitals (LCP, FID, CLS, FCP, TTFB) from actual visitors across devices/networks/locations. Real user monitoring (RUM) essential for SEO (Google uses field data for ranking) and detecting performance issues that lab tests miss.

**Proposed Implementation:**
- Client-side: `app/components/FieldPerformanceCollector.tsx` — uses `web-vitals` library to measure page loads and POST to API
- API: `app/api/field-performance/route.ts` — accepts metrics submissions, stores to `.hermes/memory/field-performance/raw/YYYY-MM-DD.jsonl`
- Aggregator: `automation/field-performance-aggregator.cjs` — daily 11:00 UTC; computes p75/p90 per metric/page; compares to 7d baseline; detects regressions (>10% degradation); thresholds: LCP p75 >2.5s warning / >4s critical, FID >100ms / >300ms, CLS >0.1 / >0.25
- Alerts: Daily Telegram summary (top 10 worst pages, trends); GitHub issue if >5% pages critical or metric degradation trend
- State retention: 90 days aggregates; 30 days raw submissions
- Privacy-safe: no PII sent; strips URL query params before storing
- Dependencies: `web-vitals` (already used in Lighthouse monitoring)
- Cost: 100% free & self-hosted; no external RUM SaaS

**Why now:** Completes performance monitoring — you have lab data (Lighthouse) and need field data to understand real user experience and SEO impact. This closes the visibility loop.
